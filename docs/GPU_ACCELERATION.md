# 🎮 GPU Acceleration Guide ## Supported Hardware - Primary target: Adreno 6xx/7xx series (KGSL path). This repo targets **OpenGL ES / Vulkan via Zink/Turnip** with the goal of hardware-accelerated rendering inside a Debian chroot on Android/Termux. ## Key Components | Component | Role | | --- | --- | | Mesa Zink | OpenGL-over-Vulkan translation layer | | Turnip (freedreno) | Vulkan driver for Adreno over KGSL | | VirGL bridge | Host-side GPU proxy for chroot clients | | Termux:X11 | Display server binding on Android | | `99-hardware-acceleration.sh` | Environment profile in the chroot | ## Probe Checklist 1. Verify KGSL device exists: ```bash ls -l /dev/kgsl-3d0 ``` 2. Confirm VirGL test socket: ```bash ls -l /data/data/com.termux/files/usr/tmp/.virgl_test ``` 3. Run `gpu-audit.sh` from the host dashboard or directly: ```bash bash ~/Systemless-Host-Termux-SU/scripts/gpu-audit.sh ``` ## Common Fixes - If Zink reports DRM-only, ensure `mount-debian.sh` is active and `/dev/dri` nodes are exposed. - If the compositor races with X11 startup, keep the `dbus-launch --sh-syntax` sequence from `user-session.sh`. - If rendering falls back to software, confirm `LIBGL_ALWAYS_HW=1` is exported and `GALLIUM_DRIVER=virgl` is set in the chroot session. ## Security Notes GPU access gives guest processes broad device access. Keep these principles: - Restrict socket access to `Ruusian5` and required Termux runtime groups. - Avoid `777` on X11/VirGL sockets in production; prefer ACLs or group policies. ## Related Docs - [ARCHITECTURE.md](../ARCHITECTURE.md) for system layout - [docs/TEST_PLAN.md](../docs/TEST_PLAN.md) for validation flows
